**核心发现：批成员地板价跨 run 几乎不变**（reg 恒 2.5k、res 恒 3.6-3.7k、fc 恒 4.4-4.5k）。
3.7k→6.8k 的谱系差异**不是批量帧本身的差异**，由批外因素决定，按贡献排序：

1. **题型构成与 calc 材质**（最大项）：calc 不进批，单价 = mixC 叙述域 3.4k/题 →

   从 4.9k/题 抬到 12.9k/题，换来 72→81 键（+9）。
3. **批大小**：共享部分（指令+判分标准+证据并集）按 1/n 摊——ins 8 题批成员 2.8-3.1k
   反而低于 fc 3 题批 4.4k（尽管 ins 证据帽 8,000 > fc 7,500）；证据帽只按 25%/题
   （SLIM4 档）～40%/题 扩容（batch.py:66-68），并集去重后实际占用低于线性。
4. **落批尾巴**：批量 run 永远混着 solo 尾巴——覆盖率制导单答（finsB r1×3=37,244，
   12.4k/题，fin_b_012/008/009 被并集挤饿后用自己证据重答）、解析失败追答 b4。
5. **模型格式服从性**：finsC 同帧同题面换 3.5-plus → 总账 +73%（b1 completion
   14,751→34,591 更啰嗦；**27 发 b4 追答 +132,770**——3.5-plus 不守【第i题】块格式，
   批量摊薄经济学当场崩塌），键面还更差（19 vs 23）。批量帧对模型格式敏感。
6. **摊销项申报差**：docsel（finsB 复用少只花 8k，routerP 全卷 31.6k）；cards2 压卡
   212,780（2.1k/题）；mixB fresh 构卡 356,165 无足够题数稀释（60 题半区摊 5.9k/题）。

单发对照锚：一次 b1 调用 ≈ 34k（p30,090+c4,117）答完 8 道 fin 选择题；同域逐题全火力
（mixB）fin 是 35.2k/题——**批量把选择题单价压到全火力的 1/8～1/10**。

## 二、批量的键面代价：哪些域挨得住，哪些域塌

### 2.1 域 × 批量带 vs 冠军件（键面均按 b_v4 键重算）

| 域 | 批量带（本册各 run） | 冠军件 | 键面代价 | 判决 |
|---|---|---|---|---|
| reg | 19-19-19-19-19，cards2=20 | 20/20@57k（cards2） | −0～−1（唯一常错 reg_b_001 为全线慢性题） | **挨得住**，且卡片帧还能满盘 |
| res | 15-18-18-15-17 | 19/20@76k（slim20） | −1～−4，方差大（res 本就是全线最高方差域，full 帧也塌 13-16） | **挨得住**（代价=方差不是偏置） |
| fc | 18-17-16-19（3题小批）；cards2 5题卡片批=12 | 18/20@98k（slim20） | −0～−2；卡片垄断信道时 −6 | **挨得住**，前提=3题小批+原文片段 |
| ins | 12(finsB)-11(finsC)-8(routerP)-8(r5)-14(mixKing)-8(cards2) | 20/20@585k（insE 全文节选卡） | **−6～−12** | **塌** |
| fin | 11-8-10-13-12-13 | 18/20@721k（full12） | **−5～−10**（选择题半塌、calc 塌） | **塌** |

### 2.2 机理（每条带一手证据）

1. **共享真实性定律**：批量摊的是共享成本；reg/res/fc 批内题真正共享文档底仓
   （同一部法规/同一份研报，题面差异是问法不是材料），并集证据≈每题所需的公共 MSIS。
   ins/fin 每题的边际 MSIS（某产品的特定条款、某公司的特定单元格）互不共享，
   并集帽只按 25%/题 扩容装不下 8 题 × 各自条款。
2. **覆盖稀释有定量法医**（batch.py:70-71, 102-107 注释即实验记录）：8 题批检索覆盖
   0.16 vs 单题 0.41（slim10 稀释类伤）；批内覆盖 <0.15 时正确率仅 48%，≥0.4 达 86%。
3. **塌方是批相关的（团灭模式）**：routerP 的 ins 8 题批
   [002,004,014,006,008,017,007,013] **整批 0/8 全错**；mixKing 同帧同批同 prompt
   重掷得 6/8。一次坏的 b1 调用带走整批——批量把逐题独立方差改造成批级相关方差，
   键面波动被批大小放大（ins 域 8↔14 的 6 键摆动就是一个批的生死）。
4. **漏检被规则放大成自信排除**（finsB run_log 一手尸检，ins_b_010）：docsel 已正确选出
   [2,4,3,16]，但批量并集证据没装进 doc2 条款，模型解题记录原文写
   "A. 国寿增益宝: 文档2中未检索到未成年人身故限制相关条款" → 规则6（补检后仍无→缺失即否定）
   把批量漏检放大成确定性错答 B（键 AB）。批量的覆盖窟窿会穿透判分规则变成系统性错。
5. **fc 挨得住的工程前提**：group_questions 对 fc 强制 mb=3（大文档小批深挖，batch.py:37）；
   cards2 用 5 题批+有损卡片替代原文 → fc 12/20（8 错里含批量帧从不错的 fc_b_002/004/009/010）。
   同一个域，批量参数（批大小×证据载体）错了照样塌。
6. **ins/fin 塌的不是"钱少"**：ins 批成员 2.8k 是全场最低价+最差键面；insE 用 585k（29k/题）
   才拿到 20/20。塌因=载体与帽，不是预算线性不足——finsB 的 fin 选择题里 facts2 接线的
   fin_b_004/012 照样对（见第三节），错的集中在慢性检索缺失题（003/005/006/007/009）。

### 2.3 批量适配域判据（可迁移 checklist）

批量安全 ⇔ 同时满足：
- ① 批内题共享文档底仓，题差在问法不在材料（reg 法规条文/res 同份研报/fc 同份合同）；
- ② 域 MSIS 是叙述性知识，批并集/卡片压缩高保真（对照：表格数值卡片压缩致命，E1 fin 7/20）；
- ③ 单题证据帽本来就小（reg base_cap 6,000），并集 25%/题 扩容不产生挤出；
- ④ 无"逐字条款裁决/多实体逐一核实/单元格取数"需求。
反判据（出现即禁批或强制配特护）：多产品对比题（ins）、成员资格扫描题（ins_b_010 类）、
单元格级取数与两笔合计陷阱（fin calc）、排序/差额计算槽。

## 三、finsB 现象：4.5k 解开 24k 重装解不开的 fin_b_004

### 3.1 账实核（一手）

- finsB：fin_b_004 = **AB ✓ @4,493 tok（p3,936 + c557）**——8 题 fin 批一次 34k b1 调用的均摊份额。
- 全档案直方图（104 个 run 目录逐一重算）：**full 帧 16 跑仅 7 对 9 错**（错法 A×4 / ABCD×2 /
  D×3 / ...），该题逐题记账 34,885～111,617/题（full13～full11）；b_mixB 逐题全火力 47,266 也错（AD）。
- **4.5k 批量帧 3.6-plus 8/8 全对**：finsB 4,493 / routerP 4,602 / r5_routerP 4,547 /
  mixKing 4,512 / router8_qB 4,640 / routerG_qB 4,578 / routerX_qB 4,685 / finslim37 5,325；
  slim 家族 4.5k 件（slim10-31）亦几乎全对。唯一批量失手 = finsC（3.5-plus，10,994，答 A）。

### 3.2 机理：摊薄反而更准的三层原因

1. **确定性覆盖替代概率性检索**：病根是速动比率必需的存货行与题面 2-gram 零重叠，
   词法打分器永远抓不到（finF 尸检）。finsB 帧的 `_RATIO_NEED` 表在题面出现"速动比率"时
   强制注入存货/流动资产/流动负债行——零成本补上 MSIS 覆盖。full 帧的 24k 花在
   r1×2/r2/仲裁的采样与复核上，**不补这块覆盖**：绑定缺失用采样预算不可替代
   （同病：full10 全卷烧 4.59M 仍治不了 fin_b_012）。
2. **绑定密度**：facts2 事实行自带 公司/年份/报表节(合并|母公司)/科目 四维身份，
   每 token 的有效信息密度远高于扁平解析的原文表格页——4.5k 里装下的是 12 个数据点
   （3 公司×2 年×3 比率）的干净绑定行。
3. **瘦上下文消灭摇摆源**：该题是方向混合陷阱（比亚迪流动比率↑但速动比率↓），full 帧
   24k+ 混合证据里布满近似数字诱饵，多轮判断在噪声下反复重掷 → 摇摆带（9/16 错，三种错法）。
   小而纯的上下文让干扰质量随瘦身下降——**"更小上下文=更少摇摆源"**（KS_MECHANISM 1.5）。
   便宜与准确不是取舍而是同源：都来自"只送必需绑定行进上下文"。

### 3.3 边界（防过度推广）

- **摊薄的合法对象只有共享成本**（指令/判分标准/同域底仓）；逐题边际 MSIS 一字不能摊。
  同一摊薄思想载体错了即翻车：E1 mega 把逐题表格摊进 900 字卡 → fin 7/20@294k。
- finsB 的 fin 总键面仍只有 11/20：facts2 只治它接了线的病（004/012），批量帧对
  慢性检索缺失题（005 两笔合计陷阱等）无解。**"4.5k 更准"是题级现象不是域级现象**——
  它证明的是"该题的 MSIS 本来就只值 4.5k"，不是"批量便宜还全面更强"。

## 四、批量线记账的坑与正确记账法

### 坑1：首题记账偏置（cards2 原型，最重的坑）

- cards2_pilot.py:82 把 5 题 mega 调用整笔记到 `qid=grp[0]`：100 题只产生 **43 个真实
  qid 行**（17 个批首题 + 26 道 calc），其余 57 题零记账。
- 账实核：mega 批 [fc_b_004, fc_b_018, fc_b_002, fc_b_010, fc_b_015] 中，ledger 只有
  fc_b_004=[7142,3167]；answer.csv 里 fc_b_004 行 12,438，其余四题各 2,129（纯卡摊销）。
- 危害：①域单价假象——cards2 ledger 算出 "fc 8 题 avg 7,744"，真实是 20 题 avg ~3.1k 答题账；
  ②审计红旗——57 行 token 与题长/答案长/难度零相关，恰是 7-24 新规明文严审项
  （"逐题 token 须与题长/答案长/摘要长相关"）。原型可用，**提交件禁用首题记账**。

### 坑2：_batch 伪键残差（整除余数）

- batch.py:162-173 的正确均摊：`usage//n` 记入每题、`_batch` 槽同步扣减防双计——
  但整除余数留在 `_batch` 伪键：mixA 13 / mixC 31 / finsB 18 / routerP 52 / mixKing 56 tok。
- 只汇总真实 qid 的分析脚本会丢掉它；totals 必须= per_qid 全键合计（含伪键）。

### 坑3：伪键不总是小钱——构卡/docsel 摊销申报差

- 同为伪键，量级差三个数量级：`_docsel_*` 残差 3~34 tok；**mixB `_digest`=356,165
  （占总账 17.8%）**；**cards2 `_card_*` 72 发=212,780（36.3%）**。
- 跨 run 比单价必须先问两件事：构卡钱在不在这本账里（见坑4）、docsel 是自跑还是复用
  （finsB 复用只花 7,976，routerP 全卷自跑 31,620）。cards2 的"reg 20/20@57k"是
  不含压卡摊销的答题账口径——同口径对比才成立。

### 坑4：缓存吃账（账不自含）

- run_b2.py:37-40 默认加载共享 digest_cache.json：本 run 用了卡但构卡钱记在**上一个** run
  （finF 998k "疑 digest 缓存被 insE 系写脏"即此类账目污染）。
- 对策：正式件 `--fresh-digests`（slimKing4 注记"账自含"）；或明确申报缓存来源与其成本。
  finsB 的 facts2 是离线词法抽取（零线上 token），属合法零成本结构件，与缓存吃账不同类。

### 坑5：批量单价≠run 单价（solo 尾巴与解析失败）

- 批量 run 永远混着贵尾巴：calc 单发（mixKing fin 单发中位 16,855）、覆盖率制导 solo
  重答（finsB r1×3=37,244）、解析失败追答 b4。**finsC 一次 27 发 b4 +132,770**——
  换模型破坏块格式服从性后，摊薄经济学整体失效。
- 正确报法分四层：批成员中位（1.3 表）/ calc 单价 / solo 尾巴 / run 均价，不可混用。

### 正确记账协议（本线已验证实现）

1. **调用时均摊**：批调用 `usage//n` 逐题入账，余数入 `_batch` 并同步扣减防双计
   （batch.py:162-173）；solo/仲裁/追答记真实 qid。
2. **伪键全保留**：构卡/docsel/批残差都留在 per_qid，`LEDGER.totals()` 全含——总账永远诚实。
3. **提交时摊平**：write_submission 把非题目键合计用 divmod 均摊到题行
   （b_schema.py:154-171），保证**逐题合计==summary**。账实核：finsB 263,758 /
   routerP 500,535 / mixKing 676,320 / cards2 586,926 四件 CSV 行和与 summary 逐一相等。
4. **resume 合并历史账**（run_b2.py:48-55）：断点续跑把旧 per_qid 与 calls 并入，杜绝漏账。

## 五、一页结论

1. **批成员地板价是域常数**（reg 2.5k / ins 2.8-3.1k / res 3.7k / fc 4.4k / fin 4.6-5.2k，
   跨 4 个 run 不变）；run 单价谱系 3.7k→6.8k 的差异由批外因素决定：
   calc 材质与占比 > 验证栈（mixKing 的 +176k 全在 calc2/calc3）> 批大小 > solo 尾巴
   > 模型格式服从性（finsC +73%）> docsel/构卡摊销。
