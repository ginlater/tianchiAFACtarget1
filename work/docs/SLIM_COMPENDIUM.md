# SLIM线全量经验手册（b_slim3b → b_slim29，30版实证总编）

> 定位：token便宜端的键最大化。骨架 = 松散合批(batch.py) + 瘦证据帽 + 低思考预算 + 单样本(砍b2)。
> 新规分数地形下，slim的 ~425-500k 账 = token分 98-99+（500k悬崖峰顶见第三章）。
> 本册数字均出自 output/b_slim*/token_ledger.json 逐调用账与 OPTIMIZATION_LOG/SLIM_LOG 台账；
> 分域键为对 answer_b_100v10 键重评（与当期官方自答键可能±1，键版本漂移所致）。

---

## 一、降token不降分（结构级压缩，逐条：操作 / 数字 / 生效条件）

### 1.1 盲检批量化 —— 单刀最大，-58k零漂移
- **操作**：文档盲检从每题一次调用改为按域合批——域内全部题共享一份候选文档卡片，
  Qwen一次输出全部题的doc_ids精选（doc_select.py `select_docs_batch`，k_coarse=12，
  输出上限 `80*len(qs)+200`；token均摊入各qid，残差留`_docsel_{domain}`保证总量诚实）。
- **数字**：slim10→slim11 实测 docsel **100次调用/89,238 tok → 6次调用/31,645 tok（-57.6k）**；
  文档选择结果与逐题版**零漂移（DISJOINT=0）**。整版 466k→409k。
- **条件**：候选卡必须全域兜底（兜底逻辑抽`_finalize_picks`单批共用）；卡片doc_id要求模型
  原样回写（数字型id与序号歧义教训）。这是纯结构改写，无准确率代价，任何档位都该默认开。

### 1.2 松散合批 —— 证据从"每题一付"改"一批一付"
- **操作**：分组键从`(域,doc_ids精确集)`放松为`域`（AFAC_SLIM4=1），批内证据取文档并集，
  一次装填供N题共用；批证据预算按题数扩容但**次线性**（slim档 +25%/题，并集去重后实际
  占用低于线性）。
- **数字**：slim4(无合批收紧) 682k/65键 → slim5(合批) **591k/68键**——省91k同时键回升；
  r1单题路径几乎清零（299k→5k），b1承接。
- **条件**：必须配 1.3 的逐题配额与 2.5 的稀释闸门，否则合批=稀释（slim10八题批教训）。

### 1.3 指令摊薄 —— 判分标准/输出规范只付一次
- **操作**：JUDGE_STD判分标准、输出格式说明、证据前缀在批prompt里只出现一次，
  按`【第i题】`块切分逐题解析（`_parse_batch`）；slim4档示例压缩为"一对天平砝码"。
- **数字**：b1从单题 **3.5k/题 → 2.4k/题**（台账口径）；与1.2合计是slim3b(781k)→slim9(490k)
  主降幅的来源。
- **条件**：块序号必须与题目序号强制一致（解析失败走1.5兜底，不许默认值）。

### 1.4 b4追答截断 —— 兜底调用瘦身2/3
- **操作**：批解析失败题的单题追答（tag=b4）证据从全量批证据截为**前5000字**，
  输出上限400；同版把批输出上限定为**1400/题**防截断（`max_tokens=1400*n+1400`）。
- **数字**：slim13→slim14 b4账 **78,404 → 13,612（-65k）**，键73→74不降反升；
  slim14 = 422,451 tok，B线早期最优。
- **条件**：只适用于兜底/追答类调用（信息需求是"补一个字母"不是"重新推理"）；
  主答b1的证据不能这么砍。

### 1.5 重试只救空白 —— 宽触发重试在瘦预算下是负资产
- **操作**：calc升级重试触发条件从"报缺数∨答案缺失∨槽位不合法"收窄为
  **"报缺数∨完全空白"**（calc.py `need_retry = (ms or not a1) if slim4 else ...`）；
  槽位小瑕疵不再触发重跑。
- **数字**：slim8宽触发 calc1b烧到65,805（较slim5 +47k）键反跌；slim9收窄后 calc1b
  **65.8k→38.1k**，整版564k→490k，键持平。
- **条件**：前提是解析器兜底已禁默认值（默认"A"实测8/8全错）——"少重试"与"瞎兜底"
  必须分开：空白必须真实追答，瑕疵不值得重跑。

### 1.6 附：参数级小刀（边际递减，打包使用）
- 思考预算阶梯：slim批b1=1300、calc1=1100/升级1600；法规域关思考20/20无损-35%。
- digest瘦身：构卡输出1200(vs1800)、装填帽8000(vs11500)；证据帽分域定制
  （无卡时 fc/fin=7500、res=6000、其余4800——slim6教训：砍卡后帽不给足=漏选爆发）。
- slim默认砍b2复核轮（单样本），仲裁b3仅批内r1≠r2时触发。
- 教训：**单纯砍帽砍思考总在±3方差带里打转还掉键**（slim4收紧-5键）；
  结构级压缩(1.1-1.5)>>参数级收紧，先动结构再拧参数。

## 二、提分小代价（检索层修复为主——slim的分全在这里）

### 2.1 限域检索扩展 —— +3键/+2.6k tok（本线性价比之王）
- **操作**：分红两笔口径的SYN同义扩展（"全年"→"全年 中期 末期"、"现金分红"→含中期分红/
  利润分配）与年报专用LEXICON（中期分红/权益乘数/少数股东权益）**挂domain条件只在
  financial_reports生效**（answerer.py L344-360）。
- **数字**：slim16全域版是灾难——"全年"污染保险域查询 **ins 13→6**，整版64键；
  限域后slim17 **77键@425,047**（vs slim14 74@422,451：**+3键仅+2.6k tok**），
  自答分首超全火力线（75.04>74.89）。
- **条件**：**SYN/LEXICON新增一律挂域**。高频词的全域扩展=查询污染，此教训已回流full线。

### 2.2 fc小批深挖 —— 分域批量，fc +4键
- **操作**：批量大小按域适配：`{"financial_contracts": 3}.get(dom, 8)`——fc域3题小批，
  其余域8题大批（batch.py `group_questions`）。
- **数字**：fc从slim17的14 → slim20的**18/20（+4，@97.8k域账）**，同版res19也创纪录；
  但slim20把ins也砍成小批→ins崩到9，slim21回配方(fc=3,ins=8)后整体77。
- **条件**：**fc与ins需求相反**（见3.3适配律）；小批只给"大文档、题间独立、需深挖"的域。

### 2.3 覆盖率制导定向单答 —— 运行时自检"被挤饿"再补刀，限fc/fin
- **操作**：批证据装填后计算每题自有证据块在并集中的存活率，**覆盖<0.5的题（每批限3）**
  用自己的完整证据solo重答，覆盖答案（batch.py L101-107, L242-263）。
- **数字**：法医量化定标——**批内覆盖<0.15的题正确率仅48%，≥0.4达86%**；
  slim19此刀全域开：fc 14→17兑现(+3)，但**ins 14→6崩**（solo小帽装不下4份条款）；
  限fc/fin域后成为slim20/21标准件。
- **条件**：只限大文档少题域(fc/fin)；ins的solo小帽反而饿死。本质是"批量稀释的运行时
  保险丝"，阈值0.5、每批3题是实测平衡点。

### 2.4 选项实体反查 —— 纯词法，res +2~4键
- **操作**：从选项抽专名短语逐文档字面反查（ENT_PROBE，answerer.py L399-416）：
  命中块若已在候选池则零成本置保护+置顶；新块只取**240字窗口**入证据，每题上限4块。
- **数字**：res_b_002/008/018/020类伤（选项专名句字面存在但被长查询稀释）；
  slim12→13 res **15→18**；与2.5合刀贡献slim13的+7键（66→73）。
- **条件**：纯词法合规零调用成本；限有选项的域（ENT_PROBE_DOMAINS）。窗口而非整块
  是防挤帽的关键。

### 2.5 稀释修复闸门 —— 逐题配额检索 + 并集总量闸门
- **操作**：批检索改为**每题独立配额检索(`per_cap=max(1500,cap//n)`)后取并集**
  （消灭伪题合并的查询前缀污染）；并集层再设总预算闸门：保护块优先装填、
  非保护块超`cap*1.15`丢弃、**硬顶`cap*1.6`保护块也不得无限叠加**（batch.py L72-95）。
- **数字**：批检索覆盖率 **0.16 → 0.41**；slim12只有配额没有闸门→保护豁免×8题叠加爆预算
  (576k)；slim13加闸门 **65→73键 @503k**——本线最大单版提分。
- **条件**：凡共享上下文皆需"per-成员配额+总量闸门"双件套，缺一即回归
  （只配额=爆账，只闸门=稀释）。

### 2.6 附：其他零成本键
- 解析失败禁默认值：默认"A"实测8/8全错，一律改最小成本真实追答（1.4的b4）。
- 判分规则化=零token键：日期格式/口径词严格核算/全年分红两笔合计/免责列举禁概括/
  数值主张现场重算（规则14/15），slim与full同享。
- 跨文档BM25按各档top1归一（公司名IDF=0病，retrieval.py `search_docs`）、
  每文档保底1块+目录块降权、forced关键词k=2且优先带数字行防样板假阳性。

## 三、权衡定理（结构性认知，供全局优化者直接引用）

### 3.1 slim错误模式 = 证据饿死，判断层无药
全部"判断层武器"在slim档三试三败：
- **同证据投票**：slim22 fin批内三采样，+118k只换+1键（77→78）——摇摆题投票收敛到错误众数；
- **异构二审**：slim23嫁接3.5-plus跨代b2/b3（full11的+5武器），+295k异构账，键仍77
  ——**异构治摇摆不治饿死**；slim29再试3.6主+3.7max复核，+476k，键反落72；
- **特护**：fin_b_012单题砸21.3k仍错——知识缺失砸钱无用。
证据不在上下文里，任何复核/投票/换模都是在错误证据上重复推理。slim的每一分都要从
检索层(第二章)拿，判断层预算应压到最低（这正是slim砍b2的合法性来源）。

### 3.2 键顶77-78的结构原因
slim17/21/22/24/25五个不同配方全部停在77-78：判断层配置迥异（有无卡/有无投票/有无异构）
而键不动，证明**约束在瘦证据帽本身**——每题~4-8k字证据装不下长尾题的关键块，
漏检集合基本固定（fin取数深井、ins多条款交叉、res跨文档比对）。
对照组：full线同期检索代码相同、帽×2.5，键90-95。→ 结论：77-78不是调参平台而是
**预算-召回曲线的物理点**；要破必须给特定题加预算（题级分配，见4.1），
全域加帽则退化成full。广度型部件同理：**卡/多查询/多文档在瘦帽下必稀释**
（slim15卡回归-7、slim18双刀-13、slim24卡+ins帽8000修复后77@675k仍不敌slim17的77@425k
——保险卡slim三试盖棺：同键多花250k）。部件收益必须在目标配置里验证，不能移植结论。

### 3.3 批量大小分域适配律（fc=3 / ins=8，需求相反）
- **fc（大文档单题深挖）**：批大=每题配额小=关键条款挤出。小批3题：fc 14→18。
- **ins（多产品条款同场对比）**：题目要求4份条款并排比较，大批的并集证据恰好是
  它要的工作台。小批/solo反而拆散并集：slim19 solo ins 14→6，slim20小批 ins→9，回8题批→14。
- 通式：**批量大小由"题的证据形状"决定**——独立深井型域要小批，交叉对比型域要大批；
  reg/res/fin居中吃8无碍。这条已蒸馏进`group_questions`的域表，mix线选块器配方同源。

### 3.4 代际适配律（slim帧对模型换代高度敏感）
slim主答代际终榜：**qwen3.6-plus(77-78) > 3.7-max(74@494k) > 3.7-plus(69@542k) > 3.5-plus(51@859k)**。
- 3.6是调校基准：全部提示词/解析器/帽都按它整定，换代=风格失配（3.7家r1b/b4兜底率
  显著升高，3.5帧内b4暴涨334k）。
- **例外：fin域3.7家拿slim史高13/20**（slim26/28两证）——薄证据下3.7敢补位推断
  （高风险高收益），3.6谨慎漏选。但fin专项slim帧3.7（finslim37）12/20@200k证伪：
  **fin深井需full级取数，slim帧任何模型都够不着**。
- 推论：换代红利是**分域互补**不是整体替换；slim帧换主模型前必须重整定帽与解析器，
  否则账涨键跌双输（slim27/29双证）。

### 3.5 分数地形注记（为什么slim线值得存在）
新规token两段曲线：<500k段递增(actual/500k×100)、>500k断崖递减((5M-t)/5M×100)，
**峰顶=499,999≈100分**。slim17形态（77键@425k）+精简四段推理合批=492,566（token分98.5）；
lean99b精确装填至498,963留1k边际。slim线的零件价值同样兑现：fc18@98k、res19@76k(slim20)、
reg19@70k(mixA)全部进入mix拼装。**slim不是降级版full，是token分曲线峰顶的占位器与零件厂。**

## 四、未竟之地（slim没试过的方向，供全局优化者接棒）

### 4.1 题级动态帽（打破3.2的物理点）
现状：帽全部是域级静态值，覆盖率制导只是"事后保险丝"。未试：**事前按题分配预算**——
用零成本词法信号（文档数、选项专名数、是否取数题、题干长度）给每题定制cap，
从全局500k池里给fin深井/ins交叉题拨6-10k、给reg拨2k。键顶77-78的漏检集合是固定的
（约10-15题），精确喂饱它们理论上是"+5键/+50k"级别的交易，比全域加帽便宜一个量级。
已排队的"ins中间形态"（单题模式+条款卡+slim级帽~300k档）是此方向的第一个具体件。

### 4.2 检索两阶段粗精（伪相关反馈，纯词法合规）
现状：单遍BM25+静态查询扩展，实体反查也是静态词表。未试：**第一遍检索的top块提取
高IDF词回灌第二遍查询**（经典PRF，零模型调用、零合规风险）；或花~200 tok让主模型在
b1里顺带输出"缺什么证据"的关键词行，触发一次定向补检后追答——把2.3的覆盖率保险丝
升级为"模型自报饿"的闭环。slim的错=饿死(3.1)，这是唯一直接对症且未开采的检索层矿脉。

### 4.3 上下文复用/前缀缓存
现状：同域各批、b1与b4、批与仲裁之间证据前缀大量重复，每次全价付prompt。未试：
①DashScope前缀缓存计费核实（若缓存命中折价，同域批排序共享前缀可再省20-30%的b1账）；
②批间增量会话：域内第二批只追加新增文档块，复用已装填的digest段；③evidence缓存
跨版本复用（digest已缓存但检索结果每版重算，调参实验的迭代成本可减半）。
风险点：平台token计量口径是否认缓存——需先用1题实测计量再上量。

### 4.4 附：一个未验证的组合猜想
3.4的fin例外(3.7敢补位) × 4.1的题级帽 → **单run内分域换模+分题配帽**：
fin批走3.7-max+8k帽，其余走3.6+现帽。mix线在拼装层做到过等效物（每域取冠军件），
但单次连贯运行的原生版从未跑过——它是"可审计单跑"叙事下逼近mix分数的唯一路径。
