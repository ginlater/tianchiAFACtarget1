# b_hex / router6-v2 独立观察报告（7-24）

观察员视角：以"前十名里最干净"为目标，对已落地 92.6902 的 b_hex 与升级弹 router6-v2
做复现、账本、推理一致性、合规暴露面、峰顶算术五项审查。全部结论可用本文所列命令复验。

实测闭环确认：b_hex summary 总账 577,459 → T=(5,000,000−577,459)/5,000,000=88.4508，
与解码 92.6902 = 99×0.5 + 85×0.3 + 88.45×0.2 完全吻合（≥500k 段平台用旧公式实锤）。

## 一、五项审查结论

### 1. 复现性 — 通过（字节级）
- `assemble_hex.py` 连跑两遍 + 与落地件三方对比：5 个文件 SHA256 全部一致
  （answer.csv=814c5b67…、token_ledger=25ed7336…）。
- `assemble_router6.py` 同样两遍字节级一致，且与当前落地件一致（answer.csv=4754079b…）。
- 全部输入（6 源 run 档案、assignment_final.json、reasonings_probe/reason_lean20/
  reasoning_probe_ledger、hex_revert_list、res005_2227_reason、b_v4 键）均在库且 git 追踪。
- 小瑕疵：assemble_hex.py 文档串写"五源"，代码 SOURCES 实为 6 个（含 b_full11，实际 0 件入选）。

### 2. 账本完整性 — b_hex 通过；router6-v2 两处硬伤
- 两件 CSV：100 行逐题 p+c=t 零违例；summary 三列 = 各列 Σ；CSV 与 token_ledger.json 逐题一致。
- b_hex 溯源：100 题的 piece 在来源 run 的 per_qid 全部真实存在；逐题账 =
  源账 + int(oh) 摊销 + 推理生成账，逐题重算零偏差；答案与源 run canon 一致零偏差。
- **router6-v2 硬伤 A（合成账）**：fin_b_014/016/019 三行标 `repriced:true`，申报账均为
  [4989,880]——该数值在**全部 run 档案中不存在**（全库扫描 0 命中）。真实源账
  （b_cards2）分别为 10,095/10,458/9,050，合计低报 **11,996** tokens。若按真账重算，
  总账 = 511,118 > 500k，峰顶位置不复存在。
- **router6-v2 硬伤 B（res_b_005 归因错乱）**：answers/token_ledger 用 b_slim23 的 22.27 真件
  （run_log 确认 final=22.27，账 [2467,2728]+推理 [1251,559] 加和无误），但
  piece_sources.json 仍写 b_slim26（脚本第 79 行从 asg 取 run 未随覆盖更新），
  evidence.json（7-23 旧档）该题仍是 b_slim26 的 **22.19** 证据链——同一提交目录内
  答案 22.27% 与证据 22.19 自相矛盾。

### 3. 推理-答案一致性 — 12 题抽查 11 题字面一致，res_b_012 一题带伤
- 抽查 res_b_005(22.27 新件)、fc_b_001/003/005、fin_b_015/016/020、reg_b_003/004/018、
  ins_b_001、res_b_012：除 res_b_012 外，答案每个槽位均在推理文中字面出现，结论句与
  answers 一致；日期三题（reg_b_003/004/018）推导链完整。
- res_b_005 新件：推导（56−45.8)/45.8=22.27% 自洽，与 b_slim23 真跑答案一致；判"596.0GWh
  为量级笔误"是模型自身论证，可辩护。
- **res_b_012**：b_hex 版结论"670,513.5**万人**"（单位笔误，实为"人"=67.05 万）与答案 67.10
  字面不一致；router6 版打补丁"最终确定约为 67.10 万人（注：原题答案单位隐含转换…）"——
  670,513.5135 万人→67.10 万人在算术上不成立，且"**原题答案**"四字暴露答案先知式写作
  （恰好绕过自检毒素正则）。LLM judge 与人工审计都可能扣这行。

### 4. 合规暴露面 — 见第二节清单（核心：键裁判选件、合成账、单题翻转 republish 模式）

### 5. 峰顶装填 499,122 算术审查
- CSV 内部算术零风险：Σ(total 列)=Σp+Σc=summary total=499,122，平台三种重算口径结果相同；
  整数账无舍入缝隙。距 500k 边界余量 878 tokens。
- **真正的风险不在 CSV 算术，在两处**：
  1) 499,122 建立在 11,996 的合成低报之上（见 2-A）；审计按档案重算即 >500k。
  2) 曲线假设单边未证实：实测只闭环了 ≥500k 段（T88.45@577,459，旧公式）；<500k 段
     "T=tot/500k×100"是对新规文本的解读，**无实测点支撑**，且新规文本"[500k,5M]=满分100"
     已被 T88.45 实测证伪——公告规则≠平台行为。情景表（对最终分的 0.2 权重贡献）：
     | 情景 | T(499,122) | 对比 b_hex 88.45 |
     | 两段假设成立 | 99.82 | +2.27 分 |
     | 旧公式全域 | 90.02 | +0.31 分 |
     | 90×tot/500k 扣分读法 | 89.84 | +0.28 分 |
     即下行有底（≈+0.28）但上行 +2.27 依赖未证实段；而合成账被查实属取消资格级,期望值为负。
