一、总体评分规则
本次测评包含三个维度：答案准确率（accuracy）、推理过程分和 Token 效率分。最终分数按以下加权公式计算：

总分 = acc 分 x 0.6 + 推理过程分 x 0.2 + Token 效率分 x 0.2
评分体系用于综合评估答案准确性、推理过程质量和 token 使用合理性，降低通过未授权方式获取答案、极低 token 消耗、缺少有效推理过程等异常提交方式获得高分的风险。

二、参赛提交格式
参赛者应按题目逐行提交结果。CSV 至少应包含以下字段：

qid,answer1,answer2,answer3,answer4,prompt_tokens,completion_tokens,total_tokens,reasoning
字段说明：

字段	说明
qid	题目编号；普通题目行应填写对应题号，汇总行可填写 summary
answer1 - answer4	答案列，用于兼容多空或多答案题目；无对应答案时可留空
reasoning	推理摘要，用于说明关键依据、计算步骤或判断理由
prompt_tokens	模型 API 原始返回的输入 token 数
completion_tokens	模型 API 原始返回的输出 token 数
total_tokens	模型 API 原始返回的总 token 数
2.1 普通题目行
每道题应提交一行结果。若题目包含多个答案，应按顺序填写至 answer1 - answer4。若题目答案数量少于 4 个，多余答案列可留空。

普通题目行中的 token usage 应记录该题所有相关模型调用的 token 总和。

2.2 summary 行
提交文件建议包含一行 summary 汇总行，用于记录整份提交的 token usage 汇总值：

summary,,,,,prompt_tokens_sum,completion_tokens_sum,total_tokens_sum,
summary 行中的 prompt_tokens、completion_tokens、total_tokens 应分别等于所有普通题目行对应字段的求和结果。若提交文件缺少 summary 行，测评脚本可根据逐题 token 字段汇总；若逐题字段与 summary 行不一致，出题方可按一致性审计规则处理。

三、允许使用的模型 API 服务
允许调用出题方指定的模型 API 服务：Qwen3.6/Qwen3.5 系列全部模型。

参赛者提交的 prompt_tokens、completion_tokens 和 total_tokens 必须直接来源于上述允许模型/API 服务返回的原始 usage 字段。

不得自行估算 token usage，不得手动修改 token usage，不得使用其他 tokenizer 重新计算后替代 API 返回值，不得遗漏中间调用，不得仅提交最后一次调用的 token usage。

若同一道题包含多次模型调用，包括但不限于草稿生成、推理、验证、纠错、重试、摘要、改写、格式整理、答案选择等步骤，该题 token usage 应为所有相关模型调用的原始 usage 逐项求和结果：

prompt_tokens = sum(prompt_tokens_k)
completion_tokens = sum(completion_tokens_k)
total_tokens = sum(total_tokens_k)
其中 k 表示该题所有与最终提交内容相关的模型 API 调用。凡是为生成最终提交内容而发生的模型调用，其 API 原始 usage 均必须计入对应题目的 token usage。生成 reasoning 过程调用模型所消耗的 token 也必须计入该题的 total token 中。未计入的调用视为未申报调用。未计入的调用视为未申报调用。

四、推理过程评分规则
4.1 评分方式
推理过程评分使用阿里云百练 DashScope API 作为 LLM judge，模型为出题方指定的模型 API。评估时仅检查选手提交的推理文本写作质量，不加载任何外部题目、原文或答案数据。

4.2 评分维度
维度	英文 key	评估内容
逻辑连贯性	logical	推理步骤之间是否存在清晰因果关系，整体链条是否自洽
论证完整性	completeness	是否具备完整分析过程，如定位、提取、推导和结论
表达清晰度	clarity	是否条理清晰、结构化、表达准确
最终推理过程分为三个维度的算术平均值。所有题目的推理过程分再取算术平均值，作为整份提交的推理过程分。

4.3 评分标准
分数区间	标准
80 分以上	逻辑清晰、论证完整、表达专业
60-79 分	有明确分析步骤和推导过程
30-59 分	有部分分析，但不完整
0-29 分	空泛、模板化或无实质内容
4.4 特殊规则
情况	处理方式
reasoning 为空或长度小于 20 字	直接记 0 分，不调用 judge API
judge API 网络异常或解析失败	该题推理过程分记 0 分，不影响整体流程
reasoning 与答案、题目或申报 token 明显不一致	可按异常处理与处罚规则处理
五、Token 效率评分规则
Token 效率分从选手提交文件读取 prompt_tokens + completion_tokens 总量，并按以下区间打分：

Token 总量	得分	说明
0	0	无 token 数据
1 - 499,999	线性递增	score = actual / 500000 x 100，例如 100 token 约为 0.02 分
500,000 - 5,000,000	100	合理区间
5,000,001 - 10,000,000	线性递减	score = 100 x (1 - (actual - 5000000) / 5000000)
大于 10,000,000	0	过量消耗
Token 总量优先使用 summary 行；若无 summary 行，可由普通题目行求和得到。若两者不一致，出题方可结合逐题记录、summary 行和结果内容进行审计。

六、一致性审计要求
参赛者提交的 prompt_tokens、completion_tokens 和 total_tokens 应真实、完整、逐题记录模型完成答题所消耗的全部 token。token usage 必须能够合理支持题目读取、答案生成、推理摘要生成以及任何中间分析、重试、改写、总结、格式化等过程。

出题方将根据题目文本长度、答案长度、推理摘要长度、题型复杂度及必要解题步骤，估算每道题的最低合理 token 使用量。

以下情况可能被标记为 token usage 异常：

prompt_tokens 低于题目文本估算 token 数；
completion_tokens 低于 answer 与 reasoning 的估算 token 数；
total_tokens 不等于 prompt_tokens 与 completion_tokens 之和；
total_tokens 低于出题方设定的最低合理 token 下限；
reasoning 的长度、信息密度或推理步骤明显超出申报 completion_tokens 可支持范围；
reasoning 包含 answer 之外的新增依据、纠错痕迹、二次总结痕迹或与模型一次输出不一致的内容，但未申报相应 token；
多道题 token usage 与题目长度、答案长度、摘要长度或题型复杂度明显不相关；
大量题目的 token usage 数值重复、过于规律、异常集中或显著低于同类题正常水平；
申报 token usage 无法支持其提交结果的完整生成过程；
普通题目行 token 合计与 summary 行 token 汇总不一致，且无法提供合理解释。
七、推理摘要要求
reasoning 应为可审计的推理依据，应说明关键依据、计算步骤或判断理由，并应能够支持 answer。推理摘要不要求提交完整思维链，但应能体现答案来源和必要判断过程。

若出现以下任一情况，该题推理合理性分可记为 0：

摘要为空；
摘要仅重复题目或最终答案；
摘要与 answer 矛盾；
摘要与题目内容无关；
摘要包含题目未提供且无法合理推出的信息；
关键计算或判断依据明显错误；
多题摘要高度模板化，无法体现具体题目依据；
摘要内容无法由申报的 completion_tokens 合理支持；
摘要存在未申报的人工改写、模型改写或二次总结痕迹。
出题方不以主观判断单独认定某段文本是否由人工改写，而是基于结果文件的一致性、完整性、token 合理性和跨题分布异常进行审计。凡是提交内容无法被其申报 token usage 合理支持的，相应内容可被判定为不可采信。

八、格式错误与缺失字段处理
若提交文件存在以下问题，出题方可根据影响程度记为格式错误、字段缺失或提交无效：

缺少必要字段；
qid 缺失、重复或无法与题目编号对应；
token 字段为空、非数字、为负数或无法解析；
total_tokens 与 prompt_tokens + completion_tokens 不一致；
文件编码、分隔符、表头或行结构导致测评脚本无法稳定读取；
提交文件实际格式与声明格式不一致，且影响测评或审计。
九、异常处理与处罚规则
对于 token usage 异常、推理摘要异常或格式异常的题目，出题方有权采取以下一种或多种处理：

按出题方估算值重算 token usage；
将该题 Token 效率分记为 0；
将该题 reasoning 判为不可采信，推理合理性分记为 0；
取消该题得分；
要求参赛者补充说明模型调用、后处理或 token 统计方式。
若整份提交中存在大量 token usage 异常、推理摘要异常、结果文件整体不自洽、模型声明不符合白名单、疑似系统性伪造 token usage、遗漏模型调用、未申报改写或异常方式加工结果文件等情形，出题方有权判定提交未通过审计，取消参赛者成绩；情节严重者取消比赛资格。

出题方可对提交结果进行以下审计：

字段完整性检查；
token usage 合理性检查；
答案与推理摘要一致性检查；
token usage 与题目长度、答案长度、摘要长度的相关性检查；
同一参赛者跨题 token 分布异常检查；
不同参赛者答案序列、错题分布和推理摘要相似性检查；
模型声明、后处理声明与 token usage 的一致性检查。
异常检测结果不必单独证明参赛者使用了非白名单模型或人工改写。若结果文件无法合理支持其提交内容来源，出题方即可按本规则对相关题目或整份提交进行处理。

十、参赛者提交声明
参赛者提交 CSV 文件即视为确认：

所有答案均由比赛规则允许的白名单模型和方法生成；
所有 token usage 均来源于白名单模型 API 原始返回的 usage 字段；
所有与最终提交内容相关的模型调用均已计入对应题目的 token usage；
未使用未授权模型、未授权外部工具或未申报方式对结果文件进行实质性改写、补全、纠错或优化。
若上述声明与结果文件内容不一致，或结果文件无法通过一致性审计，出题方有权按本规则进行处理。
